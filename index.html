Library vs Framework — what is the core difference?
Beginner
▼
Definition
A library is code you call; a framework calls your code. The key concept is Inversion of Control.
Deep Explanation
With a library (like lodash, axios), you control the program flow. You import it and call its functions whenever you want.

With a framework (like Angular, Next.js), the framework owns the control flow. It calls your code at prescribed points (lifecycle hooks, route handlers). This is the Hollywood Principle: "Don't call us, we'll call you."

Framework = opinionated scaffolding + enforced conventions. Library = focused utility, you integrate it however you want.
Simple Example
// Library: YOU call the function import axios from 'axios'; axios.get('/api/data'); // You decide when + how // Framework: IT calls YOUR function // Angular's lifecycle hook — Angular calls this, not you ngOnInit() { this.loadData(); }
Common Pitfalls
Saying "React is a framework" — React is a UI library. It only handles the View layer.
Confusing "opinionated" with "framework" — a library can be opinionated too.
Interview One-Liner
"In a library, you call the code. In a framework, the framework calls your code — this is Inversion of Control."
Interviewer Follow-Up

Q: "So if React is just a library, what makes a 'React framework'?"

Expected answer: Next.js, Remix, Gatsby — they add routing, SSR, file-based conventions ON TOP of React, making React + framework = full-stack framework.

What is an SPA? Advantages and disadvantages? ⭐
⭐Beginner
▼
Definition
A Single Page Application loads one HTML page and dynamically updates content via JavaScript without full page reloads. Navigation is handled client-side.
Deep Explanation
In a traditional MPA (Multi-Page App), every route change triggers a new HTML file from the server. In an SPA, the server sends one HTML file once; React's router intercepts navigation and swaps components in the DOM.

This gives a native app feel — no white flash between pages, shared state persists across routes, and assets (CSS, JS) only load once.

What is the Virtual DOM? How does React use it? ⭐
⭐Intermediate
▼
Definition
The Virtual DOM (VDOM) is a lightweight in-memory JavaScript representation of the real DOM. React uses it to compute the minimal set of changes needed before touching the actual DOM.
Deep Explanation
Real DOM manipulation is expensive — browsers must recalculate styles, layout, and repaint. React avoids unnecessary DOM operations with a 3-step process:

1. Render phase: When state changes, React creates a new VDOM tree.
2. Diffing (Reconciliation): React compares the new VDOM with the previous one (the "diff"). It identifies only what changed.
3. Commit phase: React applies only the minimal DOM mutations needed.

Key: The VDOM itself isn't magic — calling DOM APIs directly can be faster. The VDOM's value is in batching updates and giving React a declarative API where you describe UI by state, not by manual DOM imperatives.
Simple Example
// Without VDOM: manual DOM, expensive document.getElementById('count').textContent = count; // With React VDOM: declarative, React handles the diff const App = () => <p>{count}</p>; // React figures out only the text node changed
Common Pitfalls
Saying "VDOM is always faster than real DOM" — it's not. Direct DOM can be faster; VDOM's benefit is developer ergonomics + smart batching.
Confusing VDOM with Shadow DOM (browser native, for web components).
Interview One-Liner
"The VDOM is React's in-memory DOM copy. On state change, React diffs old vs new VDOM and only updates what changed in the real DOM — minimizing expensive repaints."
Interviewer Trap
Trap: "Is the Virtual DOM always faster?"
→ No! Svelte (no VDOM) and SolidJS are often faster because they compile to precise DOM updates. The VDOM is a useful abstraction, not a performance guarantee.

Explain React's Reconciliation (diffing) algorithm. How do keys affect it? ⭐
⭐Intermediate
▼
Definition
Reconciliation is React's algorithm for computing the diff between the current and new VDOM trees, then updating the real DOM minimally. It uses two heuristics to achieve O(n) time instead of the theoretical O(n³).
Deep Explanation
Heuristic 1 — Type check: If the root element type changes (e.g. <div> → <span>), React tears down the entire subtree and builds fresh. If the type is the same, React updates attributes in-place.

Heuristic 2 — Keys: When rendering lists, React uses keys to match old vs new elements. Without keys, React assumes position-based matching — adding an element at the top causes every other element to be re-rendered. With stable keys, React knows exactly which element moved, was added, or removed.

Fiber: Modern React uses Fiber architecture — reconciliation is async and interruptible, allowing React to pause work and prioritize urgent updates (user input > background data load).
Keys Example
// BAD: index as key — list reorder causes full re-render {items.map((item, i) => <Item key={i} {...item} />)} // GOOD: stable unique key {items.map(item => <Item key={item.id} {...item} />)}
Common Pitfalls
Using Math.random() as key — generates new key every render, forcing complete re-mount.
Using array index for lists that can be reordered, filtered, or items added/removed at front.
Interview One-Liner
"React uses two heuristics — type-check and keys — to reduce diffing from O(n³) to O(n). Keys tell React which element is which across renders."

What does JSX compile to?
Beginner
▼
Definition
JSX is syntactic sugar that Babel compiles to React.createElement() calls (or in React 17+, the new JSX transform using _jsx() from react/jsx-runtime).
Example
// What you write (JSX): const el = <h1 className="title">Hello</h1>; // What Babel compiles to (old transform): const el = React.createElement( 'h1', { className: 'title' }, 'Hello' ); // React 17+ new JSX transform (no React import needed): import { jsx as _jsx } from 'react/jsx-runtime'; const el = _jsx('h1', { className: 'title', children: 'Hello' });
Interview One-Liner
"JSX compiles to React.createElement() calls. It's not magic — it's just a more readable way to write JavaScript object trees."
Interviewer Trap
Trap: "Why did older React need import React from 'react' in every file?"
→ Because JSX compiled to React.createElement(), React had to be in scope. The new JSX transform (React 17+) auto-imports the jsx runtime, removing this requirement.

What is useState? What triggers a re-render? ⭐
⭐Beginner
▼
Definition
useState is a hook that adds local state to a functional component. Calling the setter function schedules a re-render with the new state value.
Deep Explanation
React triggers a re-render when:
1. setState (useState setter) is called with a new value
2. Parent re-renders and passes new props
3. Context value changes
4. useReducer dispatch is called

Important: React uses Object.is() to compare old vs new state. If you set state to the same value (or same object reference), React skips the re-render (bail-out optimization).
Example
const [count, setCount] = useState(0); // Triggers re-render (new value) setCount(count + 1); // Does NOT trigger re-render (same value) setCount(0); // if count is already 0 // Object trap — same reference, no re-render! const [obj, setObj] = useState({ a: 1 }); obj.a = 2; setObj(obj); // SAME reference → no re-render! setObj({ ...obj }); // NEW reference → re-render ✓
Common Pitfalls
Mutating state directly and re-setting same reference — React won't re-render.
Reading state immediately after setState — state updates are async/batched.
Interview One-Liner
"useState returns a state value and a setter. Calling the setter with a new value (by Object.is) schedules a re-render. Mutating objects without new references silently breaks reactivity."
2
What is a stale closure in useEffect? How to fix it? ⭐
⭐Intermediate
▼
Definition
A stale closure occurs when a useEffect callback (or event listener inside it) captures a variable from its initial render and never sees updated values — because the effect didn't re-run.
Deep Explanation
JavaScript closures capture variables by reference at the time the function is created. When useEffect runs, it creates a closure over the current values. If dependencies aren't listed, the effect's closure holds the old values forever — even if state updates.

Fixes:
1. Add the variable to dependency array — effect re-runs when it changes
2. Use functional setState: setCount(prev => prev + 1) — doesn't need current count in closure
3. useRef — ref.current always gives the latest value without causing re-renders
Example
// STALE CLOSURE BUG const [count, setCount] = useState(0); useEffect(() => { const id = setInterval(() => { console.log(count); // Always logs 0! }, 1000); return () => clearInterval(id); }, []); // [] means count is captured at mount (stale) // FIX 1: Add to deps }, [count]); // Re-runs when count changes // FIX 2: Functional update (no closure needed) setCount(prev => prev + 1);
Interview One-Liner
"A stale closure is when useEffect captures old state because it ran with an empty dep array. Fix: add the variable to deps, or use functional setState / useRef for latest value."

What is the race condition in React data fetching? How to prevent it? ⭐
⭐Advanced
▼
Definition
A race condition occurs when multiple async fetches are triggered in sequence (e.g. user types fast), and a slower earlier response overwrites the result of a faster later response.
Deep Explanation
User types "a" → fetch for "a" starts (slow, 800ms)
User types "ab" → fetch for "ab" starts (fast, 200ms)
Fetch "ab" resolves first (200ms), state = results for "ab"
Fetch "a" resolves later (800ms), state = results for "a" ← WRONG!

The solution is to use a cleanup flag or AbortController to ignore stale responses.
Fix with ignore flag
useEffect(() => { let ignore = false; async function fetchData() { const data = await fetch(`/api/search?q=${query}`); if (!ignore) setResults(data); } fetchData(); return () => { ignore = true; }; // cleanup }, [query]);
With AbortController (modern)
useEffect(() => { const controller = new AbortController(); fetch(`/api?q=${query}`, { signal: controller.signal }) .then(r => r.json()).then(setResults) .catch(e => { if (e.name !== 'AbortError') throw e; }); return () => controller.abort(); }, [query]);
Interview One-Liner
"Use an ignore flag or AbortController in useEffect cleanup — if a new render triggered a new fetch, mark previous responses as stale and ignore them."
4
What is prop drilling? How does Context solve it? ⭐
⭐Intermediate
▼
Definition
Prop drilling is passing props through multiple intermediate components that don't need them, just to get data to a deeply nested child. Context API provides a way to share values globally without drilling.
Example
// Problem: theme drills through A → B → C App (theme) → ComponentA (theme) → ComponentB (theme) → ComponentC uses theme // Solution: Context const ThemeCtx = React.createContext('light'); // Provider at top <ThemeCtx.Provider value={theme}><App /></ThemeCtx.Provider> // Consumer anywhere deep const theme = useContext(ThemeCtx); // No drilling!
Performance Implication
Context has a key performance issue: every consumer re-renders when the context value changes, even if the specific part they use didn't change.

Fix strategies: Split context into smaller contexts, use useMemo on the context value, or use a dedicated state library like Zustand/Redux for high-frequency updates.
Interview One-Liner
"Prop drilling passes data through components that don't use it. Context solves this but causes all consumers to re-render on any change — use carefully for high-frequency state."
5
Why use functional updates: setCount(prev => prev + 1)?
Intermediate
▼
Definition
Functional updates pass a function to the setter that receives the latest state as its argument, ensuring you never operate on stale state — regardless of batching or closures.
Why it matters
// BUG: multiple updates in same event — both read count=0 setCount(count + 1); // 0 + 1 = 1 setCount(count + 1); // 0 + 1 = 1 (still reads 0!) // Final count: 1 (expected: 2) // FIX: functional update — each receives latest setCount(prev => prev + 1); // 0 → 1 setCount(prev => prev + 1); // 1 → 2 ✓
Interview One-Liner
"Use functional updates when new state depends on old state — avoids stale closure bugs and is safe with React's automatic batching."

2
When to use useReducer instead of useState? ⭐
⭐Intermediate
▼
Definition
useReducer manages state via a reducer function — a pure function that takes (state, action) and returns new state. Best for complex state logic, multiple sub-values, or when next state depends on previous.
When to Choose
Use useReducer when:
- State has multiple sub-fields that update together
- State transitions have names/types (ADD_TODO, REMOVE_TODO)
- Complex conditionals in your setState logic
- You want testable, predictable state transitions
- Next state depends on previous in complex ways

Use useState when: state is a simple value, or 2-3 independent values.
Counter with useReducer
const reducer = (state, action) => { switch (action.type) { case 'increment': return { count: state.count + 1 }; case 'decrement': return { count: state.count - 1 }; case 'reset': return { count: 0 }; default: return state; } }; const [state, dispatch] = useReducer(reducer, { count: 0 }); dispatch({ type: 'increment' });
Interview One-Liner
"Use useReducer when state logic is complex, has named transitions, or multiple sub-values — it's like a local Redux, making state changes predictable and testable."
Interviewer Follow-Up

Q: "How does useReducer relate to Redux?"

Same mental model: reducer + dispatch + actions. Redux adds a global store, middleware, DevTools, and cross-component sharing. useReducer is component-local.

Difference between useMemo and useCallback? ⭐
⭐Intermediate
▼
Definition
useMemo memoizes a computed value. useCallback memoizes a function reference. Both recompute only when dependencies change.
Comparison
useMemouseCallback
ReturnsMemoized valueMemoized function
Use caseExpensive calculationsStable function refs for child props
EquivalentuseMemo(() => fn(), deps)useMemo(() => fn, deps)
Example
// useMemo: heavy computation cached const sortedList = useMemo( () => items.sort((a, b) => a.price - b.price), [items] ); // useCallback: stable reference for child component const handleClick = useCallback(() => { dispatch({ type: 'ADD', payload: item }); }, [item, dispatch]); // Without useCallback, new fn reference each render // → React.memo child re-renders unnecessarily
Common Pitfalls
Using useMemo/useCallback everywhere "for performance" — they have overhead too. Only use when you measure a problem.
useMemo without React.memo on child — memoized function is useless if child always re-renders anyway.
Interview One-Liner
"useMemo caches a computed value; useCallback caches a function reference. Both prevent unnecessary recalculation/recreation — but only pay off when combined with React.memo or when computation is truly expensive."
4
What is referential equality and why does it matter? ⭐⭐
Intermediate
▼
Definition
Referential equality means two variables point to the exact same object in memory (===). In React, new object/function literals created on every render always fail equality checks, causing unnecessary re-renders.
Why it breaks React.memo
const Parent = () => { // New function object EVERY render const onClick = () => doSomething(); return <Child onClick={onClick} />; }; // React.memo on Child compares: onClick_old === onClick_new // → false every time → memo is useless! // FIX: useCallback gives stable reference const onClick = useCallback(() => doSomething(), []);
Interview One-Liner
"Every render creates new objects/functions — they're always 'not equal' by reference. useCallback and useMemo preserve stable references, making React.memo and useEffect deps work correctly."

Redux / Redux Toolkit

Intermediate → Advanced
1
What is Redux? What problem does it solve? ⭐
⭐Intermediate
▼
Definition
Redux is a predictable state container for JavaScript apps. It solves the problem of sharing state across many components with a single global store and strict unidirectional data flow.
Core Concepts
Three principles:
1. Single source of truth — one global store, predictable state
2. State is read-only — only dispatch actions to change state
3. Changes via pure reducers — no side effects, testable

When to use Redux vs Context:
Context: simple, low-frequency shared state (theme, user)
Redux: complex, high-frequency, cross-feature state; dev tools needed; large team
Data Flow
Component → dispatch(action) → reducer(state, action) → new state → store → component re-renders
Interview One-Liner
"Redux provides a single predictable global store where state changes happen only through dispatched actions processed by pure reducers — making state changes traceable and testable."
2
What is a slice in Redux Toolkit? How does it simplify Redux? ⭐
⭐Intermediate
▼
Definition
A slice is a collection of Redux reducer logic and actions for a single feature. createSlice auto-generates action creators and action types from reducer functions — eliminating 90% of Redux boilerplate.
Before vs After RTK
// BEFORE (classic Redux — 40+ lines) const INCREMENT = 'counter/increment'; const increment = () => ({ type: INCREMENT }); const counterReducer = (state = 0, action) => { ... }; // AFTER RTK createSlice (10 lines!) const counterSlice = createSlice({ name: 'counter', initialState: { value: 0 }, reducers: { increment: state => { state.value += 1; }, // Immer! decrement: state => { state.value -= 1; }, setBy: (state, action) => { state.value = action.payload; } } }); export const { increment, decrement, setBy } = counterSlice.actions;
Immer Under the Hood
RTK uses Immer inside reducers — you can write "mutating" code (state.value += 1) and Immer produces a new immutable state behind the scenes. This removes the need for ...spread gymnastics in every reducer.
Interview One-Liner
"createSlice auto-generates action creators + action types from reducer functions, and Immer lets you write mutating-looking code that's actually immutable."
3
What is createAsyncThunk? How does it handle loading/error states? ⭐
⭐Advanced
▼
Definition
createAsyncThunk generates a thunk that dispatches lifecycle actions: pending, fulfilled, and rejected — automatically, based on your async function's promise.
Full Example
// 1. Create the thunk export const fetchUser = createAsyncThunk( 'user/fetchById', async (userId, { rejectWithValue }) => { try { const res = await axios.get(`/api/users/${userId}`); return res.data; } catch (err) { return rejectWithValue(err.response.data); } } ); // 2. Handle in slice extraReducers: (builder) => { builder .addCase(fetchUser.pending, state => { state.status = 'loading'; }) .addCase(fetchUser.fulfilled, (state, action) => { state.status = 'succeeded'; state.user = action.payload; }) .addCase(fetchUser.rejected, (state, action) => { state.status = 'failed'; state.error = action.payload; }); }
Interview One-Liner
"createAsyncThunk wraps async logic and auto-dispatches pending/fulfilled/rejected actions — use extraReducers in your slice to handle each lifecycle state."
4
Explain Flux architecture. How does Redux implement it? What role does Immer play? ⭐
⭐Advanced
▼
Definition
Flux is an application architecture pattern for unidirectional data flow: Action → Dispatcher → Store → View. Redux is an implementation of Flux (simplified — no dispatcher, reducer is pure function).
Flux vs Redux
ConceptFluxRedux
StoresMultipleSingle store
DispatcherExplicit dispatcherBuilt into store.dispatch
State mutationStores handle mutationPure reducers (immutable)
Data flowUnidirectionalUnidirectional
Immer's Role
Immutability is core to Redux — reducers must return new state objects, not mutate the old one. Writing { ...state, user: { ...state.user, name: action.payload } } gets verbose fast.

Immer creates a "draft" proxy of state. You mutate the draft, Immer produces a structurally-shared immutable copy. RTK uses Immer automatically in all slice reducers.
Interview One-Liner
"Redux implements Flux's unidirectional data flow — Action → Reducer → Store → View. Immer makes immutable updates ergonomic by letting you write mutating code that produces immutable state."

How would you implement automatic token refresh with interceptors? ⭐⭐
Advanced
▼
The Challenge
When an access token expires, you need to:
1. Detect the 401 error
2. Call the refresh endpoint once (not multiple times if concurrent requests fail)
3. Retry all failed requests with the new token

The tricky part: if 5 requests fail simultaneously, you only want ONE refresh call. Use a promise queue to hold subsequent requests while the first refresh happens.
Production Implementation
let isRefreshing = false; let failedQueue = []; const processQueue = (error, token = null) => { failedQueue.forEach(({ resolve, reject }) => error ? reject(error) : resolve(token) ); failedQueue = []; }; api.interceptors.response.use(null, async error => { if (error.response?.status !== 401) throw error; if (isRefreshing) { // Queue this request until token refreshed return new Promise((resolve, reject) => { failedQueue.push({ resolve, reject }); }).then(token => { error.config.headers.Authorization = `Bearer ${token}`; return api(error.config); }); } isRefreshing = true; try { const { data } = await api.post('/auth/refresh'); processQueue(null, data.token); error.config.headers.Authorization = `Bearer ${data.token}`; return api(error.config); } catch (e) { processQueue(e, null); throw e; } finally { isRefreshing = false; } });
Interview One-Liner
"Use a lock flag and a queue — first 401 triggers refresh, subsequent 401s wait in queue. When refresh resolves, drain the queue and retry all failed requests."

Mock Interview — 5 Tricky Questions

Simulate real interview pressure
Answer each question mentally first, then click "Reveal Expert Answer" to compare. These questions are designed to trip up even experienced developers.
Interviewer Question 1 of 5 — Intermediate
"You call setCount(count + 1) three times in a row inside a click handler. What will count be after the click? Why?"
Answer: Count increases by only 1, not 3. In React 18+ (auto-batching), all three setCount calls in the same event handler are batched into one re-render. Since all three read the same stale value of count (e.g. 0), they all compute 0+1=1. Only the last update wins.

Fix: Use functional updates: setCount(prev => prev + 1) three times — each receives the latest queued state, so count correctly becomes 3.
Trap: Many candidates say "it will be count + 3" — they forget about batching and stale reads.
Interviewer Question 2 of 5 — Advanced
"I have a context providing an object: {'{'} user, theme {'}'}. A deeply nested component only uses theme. If only user changes, does the theme component re-render? How would you prevent it?"
Yes, it re-renders. Every context consumer re-renders when the context value changes — React doesn't know which part of the value changed. Since {'{'} user, theme {'}'} is a new object reference every time, even if theme is unchanged, all consumers re-render.

Fixes:
1. Split contexts: separate UserContext and ThemeContext — theme component only subscribes to ThemeContext.
2. Memoize value: useMemo(() => ({'{'} user, theme {'}'}), [user, theme]) — prevents new object reference if neither changed.
3. Use Zustand/Redux for user; keep ThemeContext simple.
Trap: Candidates often say "React will bail out if theme didn't change" — wrong. Context doesn't do granular diffing.
Interviewer Question 3 of 5 — Advanced
"When can useMemo actually hurt performance? Give a concrete example."
useMemo has overhead: React must store the memoized value, compare dependencies on every render, and run the comparison logic. If the computation being memoized is trivial (e.g. items.length), the memo overhead exceeds the computation cost.

Example: const count = useMemo(() => items.length, [items]) is slower than const count = items.length. Array length access is O(1); useMemo comparison is also O(1) but adds indirection.

Rule: Only memoize when: (1) computation is demonstrably slow (measure with profiler), or (2) result is passed as prop to a React.memo child. Otherwise, the cost of memoization outweighs the benefit.
Trap: "More memoization = better performance" is a common misconception. Premature optimization is a real anti-pattern.
Interviewer Question 4 of 5 — Advanced Redux
"In RTK slice reducers, you write state.user.name = action.payload. Isn't that mutating state? Won't Redux break?"
Great question — this is Immer magic. RTK wraps all slice reducers with Immer's produce() function. When your reducer runs, it doesn't receive the real state object — it receives an Immer "draft proxy." Any mutations on the draft are recorded, and Immer produces a brand-new immutable state object at the end.

So you appear to mutate, but Immer ensures the output is a completely new object with structural sharing (unchanged parts are the same reference). Redux receives a new reference → triggers re-renders correctly.

Caveat: Outside of createSlice/createReducer (e.g. in plain reducer), you must NOT mutate — Immer isn't wrapping it.
Trap: This trips up candidates who know "Redux requires immutability" but don't know how RTK/Immer works under the hood.
Interviewer Question 5 of 5 — System Design
"You're building a large e-commerce app. Product listings, cart, user auth, search filters. Would you use Context API or Redux? Walk me through your architecture decision."
Hybrid approach — this is the senior answer:

Redux (RTK) for: Cart (complex, cross-feature, frequently updated), User auth (token, permissions — needed everywhere), Product listings (server state, pagination, caching)

Context for: Theme/locale (low-frequency, rarely changes), Modal/drawer state (UI-only, not business logic)

Consider React Query/TanStack Query for: Product listings, search results — it handles caching, loading states, background refetch, and stale-while-revalidate out of the box. This removes the need for async thunks for server state entirely.

Rule of thumb: Redux for client state (cart, UI state shared across features). React Query for server/remote state. Context for rarely-changing global config.
Expected mistake: Saying "just use Redux for everything" or "just use Context for everything" without justification. Interviewers want to see trade-off thinking.

Event Loop & Non-blocking I/O

Intermediate → Advanced
01
What makes Node.js non-blocking? ⭐
⭐ CoreIntermediate
▼
Definition
Node.js is non-blocking because I/O operations (file reads, network calls, DB queries) are delegated to the OS/libuv thread pool and the main JavaScript thread continues executing — it gets a callback when the I/O completes.
Deep Explanation
Traditional servers (e.g. Java/Apache) use a thread-per-request model: each incoming connection gets its own OS thread. Threads are expensive (~1MB stack each) and block while waiting for I/O.

Node.js uses a single-threaded event loop. When you call fs.readFile():
1. Node asks libuv to handle the file read
2. libuv either uses the OS's async I/O (epoll/kqueue) or its own thread pool
3. The JS main thread continues running other code
4. When the file is ready, the callback is queued in the event loop
5. The event loop picks it up and executes the callback

Key: "Non-blocking" means the main thread is never idle waiting. It always processes the next available task.
Blocking vs Non-blocking
// BLOCKING — main thread stalls for 200ms const data = fs.readFileSync('big.txt'); // thread WAITS here console.log('after file'); // runs after // NON-BLOCKING — main thread free immediately fs.readFile('big.txt', (err, data) => { console.log('file ready'); // runs LATER }); console.log('after readFile call'); // runs FIRST
Common Pitfalls
CPU-intensive code (loops, crypto, JSON parsing of huge objects) BLOCKS the event loop — it has nowhere to delegate.
Using sync methods (readFileSync, execSync) in server request handlers kills concurrency.
Interview One-Liner
"Node.js is non-blocking because I/O is offloaded to libuv/OS — the single JS thread never waits; it processes callbacks when I/O completes. CPU-heavy work, however, still blocks it."
Interviewer Follow-Up
Q: "If Node.js is single-threaded and non-blocking, how does it handle thousands of concurrent requests?"
The event loop multiplexes: it processes callbacks of completed I/O operations rapidly. While one request waits for DB response, the thread handles another request's CPU logic. Speed comes from never idling — not from parallelism.
02
Event loop phases in detail — timers, microtasks, I/O callbacks ⭐⭐
⭐⭐ Must KnowAdvanced
▼
Definition
The event loop is a loop with 6 phases, each processing a specific queue of callbacks. After each phase (and between callbacks in some phases), microtasks (Promises, nextTick) are drained first.
Event Loop Diagram
┌─────────────────────────────────┐ │ Event Loop │ └─────────────────────────────────┘ ① timers setTimeout / setInterval callbacks ② pending I/O TCP errors, OS callbacks deferred ③ idle/prepare internal use only ④ poll fetch I/O events, run callbacks here ⑤ check setImmediate() callbacks ⑥ close cb socket.on('close') etc. Between EVERY phase: → process.nextTick queue (drained fully) → Promise microtask queue (drained fully)
Phase Details
① Timers: Executes callbacks whose setTimeout/setInterval threshold has expired. Note: the threshold is a minimum, not exact. A setTimeout(fn, 0) might not run in this pass if the poll phase took time.

④ Poll (most important): Retrieves new I/O events. If there are callbacks to run, runs them. If not, it waits here for I/O (unless setImmediate or timers are scheduled).

⑤ Check: Runs setImmediate callbacks — always after the poll phase in the same loop iteration.

Microtasks (highest priority): process.nextTick() runs before Promises, both before the next event loop phase. They can starve the event loop if called recursively.
Execution Order Example
setTimeout(() => console.log('timeout'), 0); setImmediate(() => console.log('immediate')); Promise.resolve().then(() => console.log('promise')); process.nextTick(() => console.log('nextTick')); console.log('sync'); // Output order: // sync ← synchronous, runs first // nextTick ← microtask queue 1 // promise ← microtask queue 2 // timeout ← timer phase (phase 1) // immediate ← check phase (phase 5) // (timeout vs immediate order can vary outside I/O callbacks)
Common Pitfalls
Recursive process.nextTick() starves the event loop — I/O callbacks never get to run.
Assuming setTimeout(fn, 0) runs before setImmediate — not guaranteed outside I/O context.
Interview One-Liner
"The event loop has 6 phases: timers → pending I/O → idle → poll → check → close. After each phase, nextTick and Promise microtasks drain completely. setImmediate always runs after poll; nextTick runs before any phase transition."

04
How does Node.js handle concurrency internally? ⭐⭐
⭐⭐Advanced
▼
Definition
Node.js achieves concurrency through an event-driven, callback-based model: the single JS thread runs the event loop while libuv manages a thread pool (4 threads by default) for CPU-bound system calls.
Internal Architecture
Your JS Code (Single Thread) │ ▼ Node.js (V8 + Node APIs) │ ▼ libuv ├── Event Loop ← JS callbacks run here └── Thread Pool ← 4 threads (UV_THREADPOOL_SIZE) ├── fs operations ├── dns.lookup() ├── crypto (pbkdf2, scrypt, etc.) └── user c-addons │ ▼ OS Async I/O ← epoll (Linux) / kqueue (macOS) / IOCP (Win) ├── TCP/UDP sockets ├── pipes └── network I/O ← does NOT use thread pool
Key Insight
Network I/O is handled by the OS directly — Node registers interest with epoll/kqueue and the OS notifies libuv when data is ready. Zero threads needed. This is why Node handles 10,000 concurrent HTTP connections easily.

File system and crypto use the thread pool — these OS APIs are blocking, so libuv runs them on worker threads. If you do 5 concurrent file reads with a pool of 4, the 5th waits.

CPU-intensive JS runs on the main thread — sorting a million items, parsing huge JSON, complex math — all block the event loop. Solution: Worker Threads.
Interview One-Liner
"Node handles network I/O via OS async primitives (no threads), file/crypto via libuv's 4-thread pool, and CPU work on the main JS thread. Concurrency = event loop multiplexing callbacks, not true parallelism."

Streams in Node.js — types, pipe, backpressure ⭐⭐
⭐⭐ Very ImportantAdvanced
▼
Definition
Streams are objects that let you read or write data piece-by-piece (chunks) instead of loading everything into memory. They're EventEmitter instances with a standardized interface for handling data flow.
4 Stream Types
TypeDescriptionExample
ReadableSource of data you consumefs.createReadStream(), req
WritableDestination you write tofs.createWriteStream(), res
DuplexBoth readable + writableTCP socket, net.Socket
TransformDuplex that modifies datazlib.createGzip(), crypto cipher
Why Streams Matter
// Without streams: loads ENTIRE 2GB file into memory const data = fs.readFileSync('huge.mp4'); res.end(data); // memory spike, could crash server // With streams: processes in chunks (e.g. 64KB each) // Memory usage stays constant regardless of file size! fs.createReadStream('huge.mp4').pipe(res);
Real-World: gzip streaming
const zlib = require('zlib'); // Pipe chain: file → gzip → response // Each chunk flows through without full file in memory fs.createReadStream('large-file.json') .pipe(zlib.createGzip()) // Transform stream .pipe(res); // Writable stream (HTTP response) // Or with pipeline (handles errors better): const { pipeline } = require('stream/promises'); await pipeline( fs.createReadStream('in.txt'), zlib.createGzip(), fs.createWriteStream('out.gz') );
Backpressure — Critical Concept
Problem: If a Readable produces data faster than a Writable can consume it, you get memory overflow. Example: reading a fast SSD and writing to a slow network.

Backpressure is the mechanism where the Writable signals "slow down" to the Readable:
- writable.write() returns false when internal buffer exceeds highWaterMark
- The Readable should pause until the Writable emits 'drain'
- .pipe() handles this automatically — it pauses the source on false, resumes on drain

If you implement manual streams without handling backpressure, your app will leak memory on slow clients.
Interview One-Liner
"Streams process data in chunks — constant memory regardless of file size. pipe() chains streams and handles backpressure automatically. Use pipeline() from stream/promises for proper error propagation."

The cluster module spawns multiple Node.js worker processes (one per CPU core) that all share the same server port. The master process distributes incoming connections to workers using round-robin, maximizing CPU utilization.
Architecture Diagram
Master Process (PID 1001) ├── Listens on :3000 ├── Distributes connections (round-robin) └── Manages worker lifecycle (respawn on crash) │ ├── Worker 1 (PID 1002) — CPU core 1 ├── Worker 2 (PID 1003) — CPU core 2 ├── Worker 3 (PID 1004) — CPU core 3 └── Worker 4 (PID 1005) — CPU core 4 Each worker = full Node.js process Shared: port binding, IPC channel to master NOT shared: memory, state (use Redis for shared state)
Implementation
const cluster = require('cluster'); const os = require('os'); const numCPUs = os.cpus().length; // e.g. 8 if (cluster.isPrimary) { console.log(`Master ${process.pid} — forking ${numCPUs} workers`); for (let i = 0; i < numCPUs; i++) cluster.fork(); cluster.on('exit', (worker, code, signal) => { console.log(`Worker ${worker.process.pid} died — respawning`); cluster.fork(); // auto-restart crashed workers }); } else { // Worker: each runs normal Express app const app = require('express')(); app.get('/', (req, res) => res.send(`Worker ${process.pid}`)); app.listen(3000); console.log(`Worker ${process.pid} started`); }
Pitfalls
Workers don't share memory — in-memory sessions, caches will differ per worker. Use Redis for shared state.
Stateful WebSocket connections may go to different workers on reconnect. Use sticky sessions or a pub/sub layer.
Interview One-Liner
"Cluster forks N worker processes (one per CPU core), all sharing port 3000. Master distributes connections round-robin. Workers are isolated processes — no shared memory. Use Redis for shared state."

How do you identify and prevent memory leaks in Node.js? ⭐
⭐Advanced
▼
Definition
A memory leak in Node.js occurs when objects that are no longer needed retain references that prevent V8's garbage collector from reclaiming the memory — causing heap to grow until OOM crash.
Common Causes
1. Unremoved event listeners: Attaching listeners in request handlers without removing them. Each request adds one more listener.

2. Global/closure references: Variables captured in closures that outlive their usefulness.

3. Unbounded caches: In-memory Maps/Objects that grow forever without eviction.

4. Forgotten timers: setInterval without clearInterval, holding references to outer scope.
Examples
// LEAK 1: event listener accumulation app.get('/data', (req, res) => { emitter.on('data', handler); // adds listener every request! }); // FIX: emitter.once() or removeListener when done // LEAK 2: growing cache const cache = {}; function getData(id) { cache[id] = fetchFromDB(id); // never evicted, grows forever } // FIX: use Map with size limit, or LRU cache (lru-cache npm) // Diagnose with: process.memoryUsage(); // heapUsed, heapTotal, rss // Use --inspect flag + Chrome DevTools memory profiler // Or: clinic.js, node --heapsnapshot
Interview One-Liner
"Common leaks: unremoved listeners, unbounded caches, forgotten timers. Diagnose with process.memoryUsage() and Node's --inspect + heap snapshots. WeakMap/WeakSet for caches that should allow GC."

Mock Interview — 5 Tricky Questions

Trap Detection
Answer each mentally first. These are designed to expose common misconceptions.
Interviewer Q1 · Event Loop
"Node.js is single-threaded. So how does it do anything in parallel? Is it truly single-threaded?"
Nuanced answer: Node.js is single-threaded for JavaScript execution — your JS code runs on one thread. But internally it is NOT fully single-threaded:

• libuv thread pool: 4 worker threads handle blocking I/O (file system, DNS, crypto)
• OS async I/O: epoll/kqueue handles thousands of network connections natively
• V8: has its own internal threads for GC, optimization

So the correct statement is: "Your JavaScript runs on one thread. Node.js as a platform uses multiple threads internally."
Trap: Saying "Node is completely single-threaded" is wrong. Saying "Node is multi-threaded" is also wrong. The answer requires nuance.
Trap Question Q2 · setTimeout(fn, 0)
"What does setTimeout(fn, 0) actually guarantee? Will it run immediately after the current synchronous code?"
It does NOT run immediately after sync code — there are queues between them.

After sync code finishes: process.nextTick queue drains → Promise microtasks drain → THEN the event loop's timer phase checks if setTimeout threshold passed.

setTimeout(fn, 0) means "run at earliest in the next timer phase" — but nextTick and Promises run before it:

setTimeout(() => console.log('A'), 0);
Promise.resolve().then(() => console.log('B'));
process.nextTick(() => console.log('C'));

Output: C → B → A (not A first!)
Common mistake: "setTimeout 0 runs immediately after sync code." nextTick and Promises drain first — always.
Interviewer Q3 · Streams
"Your Node.js server is reading a 5GB file and streaming it to an HTTP client. The client is on a slow connection. What can go wrong and how do you prevent it?"
Problem: Backpressure overflow. The file system reads chunks at ~500MB/s but the client receives at ~1MB/s. The Readable stream produces data far faster than the Writable (HTTP response) can consume it. Without backpressure handling, the unread chunks pile up in the Node.js process memory → memory grows → eventual OOM crash.

Wrong approach: readStream.pipe(res) — actually this IS correct! pipe() handles backpressure. If write() returns false, pipe() pauses the readable automatically and resumes on 'drain'.

What DOESN'T handle backpressure: manually reading all chunks with 'data' events and writing without checking write()'s return value.

Best approach: Use stream.pipeline(readStream, res) from stream/promises — handles backpressure AND cleans up on errors (pipe() doesn't clean up on error!).
Trap: Many candidates say "use pipe" without explaining WHY it solves backpressure, or worse, don't know that pipe() doesn't propagate stream errors properly — pipeline() does.
Trap Question Q4 · require() caching
"If I require() the same module in 10 different files, how many times does Node.js run that module's code?"
Once. Node.js caches the result in require.cache after the first load. Subsequent require() calls return the cached exports object — the module code does NOT re-execute.

This is why module-level state is shared: if module A exports a counter object and module B increments it, module C will see the incremented value.

To force re-evaluation: delete require.cache[require.resolve('./module')] before requiring again (rarely needed, but useful in testing).

ESM difference: ES modules also cache, but they use "live bindings" — the exported variable is a live reference to the module's variable, not a copy. CJS exports a snapshot at export time.
Trap: Candidates who say "it runs 10 times" misunderstand Node module system. The singleton pattern in Node.js relies entirely on this caching behavior.
Interviewer Q5 · Architecture Design
"You have a Node.js API running on a server with 8 CPU cores. A client reports that under high load, the API becomes slow even for simple requests. Walk me through how you'd diagnose and fix it."
Diagnosis approach:
1. Check CPU usage — if one core is at 100%, it's event loop blocking
2. Use --prof flag + node tick processor to find hot functions
3. Check process.memoryUsage() — memory leak can cause GC pauses
4. Use clinic doctor (clinicjs.org) for comprehensive profiling

Root causes and fixes:
• CPU-bound route (JSON.parse of huge body, crypto): offload to Worker Thread pool
• Single process bottleneck: running only 1 Node process on 8-core machine — add Cluster with 8 workers or use PM2 cluster mode
• Sync I/O in handler: replace readFileSync with async version
• Memory leak → GC pauses: profile heap, find retaining references
• Unoptimized database query: add indices, use connection pooling

The key insight: if the problem is CPU, Cluster doesn't help (each worker still blocks on CPU). Use Worker Threads for CPU isolation + Cluster for request distribution.
Expected mistake: Only suggesting Cluster without understanding that CPU blocking needs Worker Threads. Or suggesting rewriting in another language immediately without diagnosing first.

⭐ Basic Level Beginner → Intermediate
01
What is Express.js? Why use it over plain Node.js? ⭐
⭐ CoreBeginner
▼
Definition
Express.js is a minimal, unopinionated web framework for Node.js. It wraps Node's http module with a higher-level API: routing, middleware, request/response helpers, and a plugin ecosystem.
Deep Explanation
Plain Node.js http.createServer() gives you a raw callback with req and res. You manually parse URLs, query strings, request bodies, set headers, handle routing with if/else — everything from scratch.

Express adds:
Routing — app.get('/users/:id', handler) with param extraction
Middleware chain — composable, ordered functions that transform req/res
Response helpers — res.json(), res.status(404).send()
Template engines — Pug, EJS, Handlebars
Error handling — centralized 4-argument error middleware

Express adds almost no performance overhead — it's ~0.5ms per request vs raw Node. The tradeoff is worthwhile for any non-trivial application.
Side-by-Side
// Plain Node.js — 20 lines for one route const http = require('http'); const server = http.createServer((req, res) => { if (req.url === '/users' && req.method === 'GET') { res.writeHead(200, { 'Content-Type': 'application/json' }); res.end(JSON.stringify({ users: [] })); } }); // Express — clean, readable const app = express(); app.get('/users', (req, res) => res.json({ users: [] }));
Common Pitfalls
Saying "Express is faster than Node" — Express runs ON Node; it can't be faster, only slightly slower due to abstraction overhead.
Using Express for CPU-intensive work — it's a web framework, not a compute layer.
Interview One-Liner
Express wraps Node's http module with routing, middleware, and response helpers — turning 50 lines of manual parsing into 3 lines of clean code, with near-zero performance cost.
Follow-Up Question
Q: "What are alternatives to Express in Node.js and when would you choose them?"
Fastify (2x faster, JSON schema validation built-in), Koa (same team as Express, uses async/await properly), Hono (edge-runtime compatible), NestJS (opinionated, Angular-like structure for large teams). Choose Fastify for high-throughput APIs; Koa if you want Express but modern; NestJS for large enterprise apps needing structure.
02
What does app.use(express.json()) do? ⭐
⭐Beginner
▼
Definition
express.json() is a built-in middleware that parses incoming requests with a Content-Type: application/json header and populates req.body with the parsed JavaScript object.
Deep Explanation
HTTP requests carry a body as a raw string/buffer. Without a body parser, req.body is undefined. express.json() does three things:
1. Checks if Content-Type is application/json
2. Reads the raw request stream and buffers the bytes
3. Parses with JSON.parse() and assigns to req.body

This replaced the old body-parser package — it's been built into Express since v4.16. Without it, POST/PUT bodies are simply unavailable.

There's also express.urlencoded({ extended: true }) for HTML form submissions (application/x-www-form-urlencoded).
Example
// Without express.json() app.post('/data', (req, res) => { console.log(req.body); // undefined ← silent bug! }); // With express.json() — must come BEFORE routes app.use(express.json()); app.post('/data', (req, res) => { console.log(req.body); // { name: 'Alice', age: 25 } res.json({ received: req.body }); }); // With size limit (security — prevent huge payloads) app.use(express.json({ limit: '10kb' }));
Common Pitfalls
Registering express.json() AFTER your routes — middleware runs top-to-bottom; body won't be parsed for those routes.
Not setting a size limit — a 100MB JSON payload can crash your server without { limit: '10kb' }.
Forgetting Content-Type: application/json in client requests — body parser silently skips non-matching content types.
Interview One-Liner
express.json() is middleware that reads the raw request stream, JSON-parses it, and attaches the result to req.body — without it, POST body is undefined. Always register it before routes, with a size limit.
Interviewer Trap
Trap: "What if the request body isn't valid JSON but the Content-Type header says application/json?"
→ Express will return a 400 Bad Request automatically with a SyntaxError. This is actually good default behavior — you don't need to manually validate JSON syntax.
03
What is middleware? What is its function signature? ⭐
⭐ CoreBeginner
▼
Definition
Middleware is a function that has access to the request (req), response (res), and the next function in the request-response cycle. It can execute code, modify req/res, end the cycle, or call next() to pass control to the next middleware.
Deep Explanation
Express processes a request through a middleware chain — a series of functions called in the order they were registered. Each function decides:
- Do the work and call next() → pass to next middleware
- Send a response → end the chain
- Call next(err) → skip to error middleware

Four types of middleware:
1. Application-level: app.use(fn) — runs for all routes
2. Router-level: router.use(fn) — scoped to a router
3. Error-handling: (err, req, res, next) — 4 args, catches errors
4. Built-in: express.json(), express.static()
5. Third-party: helmet, cors, morgan, rate-limit
Signature Types
// Standard middleware — 3 args const logger = (req, res, next) => { console.log(`${req.method} ${req.path}`); next(); // MUST call next or send response }; // Error middleware — 4 args (must be exactly 4) const errorHandler = (err, req, res, next) => { console.error(err.stack); res.status(500).json({ error: err.message }); }; // Middleware that ends the cycle (no next) const authGuard = (req, res, next) => { if (!req.headers.authorization) { return res.status(401).json({ error: 'Unauthorized' }); } next(); }; app.use(logger); app.use('/api', authGuard); app.use(errorHandler); // always LAST
Common Pitfalls
Forgetting to call next() — request hangs indefinitely, browser times out.
Error middleware with only 3 args — Express won't treat it as error middleware. Must have exactly 4.
Sending response AND calling next() — causes "headers already sent" error.
Interview One-Liner
Middleware is (req, res, next) => void — it transforms the request, ends the cycle with res.send/json, or passes control with next(). Error middleware is (err, req, res, next) — exactly 4 arguments.
04
What does next() do? What about next(err)? ⭐
⭐Beginner
▼
Definition
next() advances to the next middleware in the chain. next(err) skips all remaining regular middleware and jumps to the nearest error-handling middleware (4-argument signature).
Deep Explanation
Express internally maintains an array of middleware functions for each route. When you call next(), Express pops the next function from the array and invokes it.

next('route') — special: skips remaining middleware in current route, moves to next route
next(err) — any truthy non-string argument: triggers error middleware
next() — moves to next regular middleware

Without calling next() or sending a response, the connection hangs until the client times out. This is a common source of "mysterious" slow requests.
Example
// next() flow app.use((req, res, next) => { req.user = 'Alice'; // attach data to req next(); // → goes to app.get handler }); app.get('/profile', (req, res) => { res.json({ user: req.user }); // 'Alice' }); // next(err) flow app.get('/data', async (req, res, next) => { try { const data = await fetchData(); res.json(data); } catch (err) { next(err); // skips to error handler } }); // Error handler catches it app.use((err, req, res, next) => { res.status(500).json({ error: err.message }); });
Common Pitfalls
Calling next() after res.json() — response already sent, next middleware may try to write headers again → crash.
Not wrapping async handlers in try/catch with next(err) — unhandled promise rejections in Express 4 don't hit error middleware automatically.
Interview One-Liner
next() advances the middleware chain. next(err) bypasses all regular middleware and jumps to error handlers (4-arg). Always either call next() or send a response — never both, never neither.

CORS

⭐ Basic Beginner → Intermediate
05
What is CORS? Why is it needed in MERN stack? ⭐
⭐ CoreBeginner
▼
Definition
CORS (Cross-Origin Resource Sharing) is a browser security mechanism that blocks web pages from making requests to a different origin (protocol + domain + port) than the one that served the page, unless the server explicitly allows it via response headers.
Deep Explanation
The browser enforces the Same-Origin Policy — your frontend at http://localhost:3000 cannot fetch from http://localhost:5000 (different port = different origin) without CORS headers.

In MERN: React runs on port 3000, Express on 5000 — always different origins. Without CORS, every API call from React will be blocked by the browser (NOT the server — the server receives and processes it, but the browser hides the response).

How it works:
1. Browser sends request with Origin: http://localhost:3000
2. Server responds with Access-Control-Allow-Origin: http://localhost:3000
3. Browser checks header and allows or blocks the response

Preflight requests: For non-simple requests (PUT, DELETE, custom headers), browser first sends an OPTIONS request. The server must respond with allowed methods/headers before the actual request proceeds.
Express CORS Setup
const cors = require('cors'); // Allow ALL origins (dev only — NEVER in production) app.use(cors()); // Production: whitelist specific origins app.use(cors({ origin: ['https://myapp.com', 'https://www.myapp.com'], methods: ['GET', 'POST', 'PUT', 'DELETE'], allowedHeaders: ['Content-Type', 'Authorization'], credentials: true, // allow cookies/auth headers maxAge: 86400 // cache preflight 24h })); // Dynamic origin (e.g. allow multiple env URLs) app.use(cors({ origin: (origin, callback) => { const allowed = process.env.ALLOWED_ORIGINS.split(','); callback(null, allowed.includes(origin)); } }));
Common Pitfalls
Using origin: '*' with credentials: true — browsers reject this combination. Cannot use wildcard with credentials.
"My CORS is broken" when actually the server is returning an error (5xx) before CORS headers are attached — fix the server error first.
Forgetting CORS only applies in browsers — Postman/curl never enforces CORS. "It works in Postman" doesn't mean CORS is configured.
Interview One-Liner
CORS is a browser-enforced policy blocking cross-origin requests. In MERN, React (port 3000) and Express (port 5000) are different origins — Express must send Access-Control-Allow-Origin headers, or browsers block the response client-side.
Interviewer Trap
Trap: "Does CORS protect the server?"
→ No! CORS is enforced by the browser, not the server. The server processes the request regardless — it's the browser that hides the response. To protect the server, use authentication, rate limiting, and IP allowlisting.

▼
Definition

// App-level: runs for EVERY request app.use(morgan('combined')); // logging all app.use(helmet()); // security headers all app.use(express.json()); // body parsing all // Router-level: scoped to /api/v1 const apiRouter = express.Router(); apiRouter.use(authenticateJWT); // only for /api/v1/* routes apiRouter.use(rateLimiter); // only for /api/v1/* routes apiRouter.get('/users', getUsers); apiRouter.post('/users', createUser); app.use('/api/v1', apiRouter); // mount router // Public routes bypass apiRouter's auth middleware app.post('/auth/login', login); // no JWT needed

08
How to implement JWT authentication as middleware? ⭐⭐
⭐⭐Intermediate
▼
Definition
JWT auth middleware extracts the token from the Authorization: Bearer <token> header, verifies its signature and expiry using a secret key, and attaches the decoded payload to req.user for downstream handlers.
Production Implementation
const jwt = require('jsonwebtoken'); const authenticateJWT = (req, res, next) => { const authHeader = req.headers['authorization']; if (!authHeader?.startsWith('Bearer ')) { return res.status(401).json({ error: 'Missing or malformed Authorization header' }); } const token = authHeader.slice(7); // remove 'Bearer ' try { const decoded = jwt.verify(token, process.env.JWT_SECRET); req.user = decoded; // { userId, email, role, iat, exp } next(); } catch (err) { if (err.name === 'TokenExpiredError') { return res.status(401).json({ error: 'Token expired' }); } res.status(403).json({ error: 'Invalid token' }); } }; // Authorization middleware (checks role) const requireRole = (role) => (req, res, next) => { if (req.user?.role !== role) { return res.status(403).json({ error: 'Forbidden' }); } next(); }; // Usage router.get('/admin/stats', authenticateJWT, requireRole('admin'), getStats);
Interview One-Liner
JWT middleware: extract token from Authorization header, verify with jwt.verify() — which checks signature AND expiry atomically — attach decoded payload to req.user, call next() or return 401/403.
09
How does Express handle the request lifecycle from app.listen() to route handler? ⭐⭐⭐
Advanced
▼
Definition
Express's request lifecycle is a sequential pipeline: TCP connection → Node http server → Express Router layer matching → middleware chain execution → route handler → response.
Full Lifecycle
1. app.listen(3000) └─ Internally calls http.createServer(app) + server.listen(3000) └─ Express app IS the request listener function 2. TCP connection arrives └─ Node's http module parses HTTP/1.1 request └─ Creates IncomingMessage (req) and ServerResponse (res) objects 3. What is JWT? What are its three parts? ⭐
⭐ CoreBeginner
▼
Definition
JWT (JSON Web Token) is a compact, URL-safe token format for transmitting claims between parties. It is self-contained — the server can verify it without a database lookup. Structure: Header.Payload.Signature
Three Parts
// A real JWT: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9 ← HEADER (base64url) .eyJ1c2VySWQiOiI2NjYiLCJlbWFpbCI6ImFAYi5jb20iLCJpYXQiOjE2MH0 ← PAYLOAD .HMACSHA256(header + '.' + payload, SECRET) ← SIGNATURE // Header (decoded) { "alg": "HS256", "typ": "JWT" } // Payload (decoded) — NEVER put passwords here! { "userId": "666", "email": "alice@example.com", "role": "admin", "iat": 1609459200, // issued at "exp": 1609462800 // expires (1 hour) } // Signature — HMAC-SHA256 of encoded header + payload + secret // Tampering payload → signature fails verification
Common Pitfalls
The payload is base64-encoded, NOT encrypted — anyone can decode it. Never store passwords, card numbers, or sensitive PII in JWT payload.
Confusing "signed" with "encrypted" — JWT signature proves authenticity (wasn't tampered with), not confidentiality (not secret).
Interview One-Liner
JWT = Header (algorithm) + Payload (claims, base64 readable) + Signature (HMAC of header+payload using secret). The server verifies by re-computing the signature — no DB lookup needed. Payload is readable, not encrypted.

Security risks of JWT. How to mitigate them? ⭐⭐
Intermediate
▼
Security Risks
alg: none attack — attacker sets algorithm to "none", removing signature verification. Fix: explicitly specify algorithm in jwt.verify(token, secret, { algorithms: ['HS256'] }).
Weak secret — brute-forceable short secrets. Fix: use a 256-bit random secret (crypto.randomBytes(32).toString('hex')).
No expiry — tokens valid forever. Fix: always set exp (e.g. 15 minutes for access tokens).
Sensitive data in payload — payload is base64, not encrypted. Fix: only store non-sensitive identifiers (userId, role).
Token theft via XSS — localStorage accessible to JS. Fix: HttpOnly cookies for refresh tokens, memory for access tokens.
Cannot revoke tokens — JWTs are stateless. Fix: short expiry + refresh token rotation + token revocation list (Redis blocklist).
Interview One-Liner
Key JWT risks: alg:none attack (fix: specify algorithms), weak secrets (fix: 256-bit random), no expiry (fix: short exp), token theft (fix: HttpOnly cookies), no revocation (fix: short-lived + Redis blocklist).

Implement refresh token flow. Token rotation and revocation? ⭐⭐⭐
Advanced
▼
Definition
Refresh token flow: a short-lived access token (15min) + long-lived refresh token (7 days). When the access token expires, the client uses the refresh token to silently obtain a new access token without re-login.
Full Flow Diagram
LOGIN POST /auth/login → Verify credentials → Issue: accessToken (15min, in response body) refreshToken (7 days, HttpOnly cookie) → Store refreshToken hash in DB (or Redis) ACCESS TOKEN EXPIRES (401) POST /auth/refresh → Read refreshToken from HttpOnly cookie → Verify it exists in DB (not revoked) → Verify jwt.verify(token, REFRESH_SECRET) → TOKEN ROTATION: Delete old refresh token from DB Issue new refreshToken (HttpOnly cookie) Issue new accessToken (response body) LOGOUT / REVOCATION POST /auth/logout → Delete refreshToken from DB → Clear cookie → (Optional) add access token to Redis blocklist until exp
Implementation
// Login: issue both tokens const login = async (req, res) => { const user = await User.verifyCredentials(req.body); const accessToken = jwt.sign( { userId: user._id, role: user.role }, process.env.JWT_SECRET, { expiresIn: '15m' } ); const refreshToken = jwt.sign( { userId: user._id }, process.env.JWT_REFRESH_SECRET, { expiresIn: '7d' } ); await RefreshToken.create({ token: hash(refreshToken), userId: user._id }); res.cookie('refreshToken', refreshToken, { httpOnly: true, secure: true, sameSite: 'strict', maxAge: 7*24*3600000 }); res.json({ accessToken }); }; // Refresh: rotate tokens const refresh = async (req, res) => { const { refreshToken } = req.cookies; if (!refreshToken) return res.status(401).end(); const stored = await RefreshToken.findOne({ token: hash(refreshToken) }); if (!stored) return res.status(403).json({ error: 'Revoked' }); const payload = jwt.verify(refreshToken, process.env.JWT_REFRESH_SECRET); await stored.deleteOne(); // ROTATE — invalidate old token // Issue new pair... };
Interview One-Liner
Refresh flow: short-lived access token + long-lived refresh token in HttpOnly cookie. On access token expiry, verify refresh token against DB, rotate it (delete old, issue new), return new access token. Revocation = delete from DB.

Mock Interview — 5 Trap Questions

Designed to expose misconceptions
Answer each question mentally before revealing. These are real interview questions with hidden traps.
Interviewer Q1 · Middleware · Intermediate
"I wrote an error-handling middleware but it never gets called, even when I throw an error in a route. Here's my code — what's wrong?"

app.use((err, req, res) => res.json({ error: err.message }));
The problem: The error middleware has only 3 parameters instead of 4. Express identifies error middleware by checking whether the function has exactly 4 parameters. With 3 args, Express treats it as regular middleware — it never receives errors.

Fix: (err, req, res, next) => res.json({ error: err.message })

The next parameter must be declared even if you don't use it. Express uses fn.length to detect middleware type — fn.length === 4 triggers error-handler mode.
Trap: Most candidates see "error middleware" and check if it's registered last. The real issue is parameter count — a subtle trap even experienced Express devs fall into.
Trap Question Q2 · CORS · Beginner/Intermediate
"I set up CORS with origin: '*' and credentials: true on my Express server. My React app sends cookies in the request but they're being ignored. Why?"
You cannot combine origin: '*' with credentials: true. The browser explicitly rejects this combination as a security rule — wildcard origin means "anyone", but credentials (cookies, auth headers) should never be sent to "anyone."

Fix 1 — Explicit origin:
cors({ origin: 'http://localhost:3000', credentials: true })

Fix 2 — Dynamic whitelist:
cors({ origin: (origin, cb) => cb(null, allowedOrigins.includes(origin)), credentials: true })

Also, the React side needs: axios.get(url, { withCredentials: true }) or fetch(url, { credentials: 'include' })
Expected mistake: Candidates blame the browser or try adding more CORS headers. The combination of wildcard + credentials is simply invalid per the CORS spec.
Interviewer Q3 · JWT Security · Advanced
"A user changes their password. Should their existing JWT stop working? How would you implement that, given JWTs are stateless?"
This exposes a fundamental JWT limitation: stateless tokens can't be revoked before expiry. If a user changes their password, their old JWT technically remains valid until it expires.

Solutions (trade-offs between statefulness and security):

1. Short expiry (1-15min) — old token becomes useless soon. Simplest but doesn't give instant revocation.

2. Token versioning in DB: Store a tokenVersion field in User. Include it in JWT payload. Increment on password change. Middleware checks: if (decoded.tokenVersion !== user.tokenVersion) return 401. Requires one DB query per request — partially stateful.

3. Redis blocklist: On password change, add old token's JTI (JWT ID) to Redis with TTL = token's remaining lifetime. Middleware checks Redis on each request.

4. Refresh token invalidation: Delete all refresh tokens for that user — they can't get new access tokens after the current one expires.
Trap: Candidates who say "just verify the JWT" miss the point — verification only checks signature, not revocation. This question tests whether you understand the stateless limitation.
Trap Question Q4 · Middleware Order · Intermediate
"My Express app has this code. A POST to /api/users returns req.body as undefined even though I'm sending JSON. Find the bug."

app.post('/api/users', (req, res) => res.json(req.body));
app.use(express.json());
Middleware order bug. Express runs middleware in registration order, top to bottom. The route handler is registered BEFORE express.json(), so when the route handler runs, body parsing hasn't happened yet — req.body is undefined.

Fix: Always register express.json() before any routes that need it:
app.use(express.json()); // ← FIRST
app.post('/api/users', (req, res) => res.json(req.body)); // ← after

This is one of the most common Express bugs for beginners — and interviewers love it because it's subtle in a code review.

Interviewer Q5 · Architecture · Senior Level
"You're building a MERN e-commerce app. The product team wants: product listing (public), cart management (auth), checkout with payment + email (auth + complex logic), and admin dashboard (admin only). Walk me through the complete Express architecture."
Senior answer — layer by layer:

Folder structure: routes/ → controllers/ → services/ → models/ + middleware/

App-level middleware: helmet() (security headers), cors() (whitelist), express.json() (body parser), morgan() (logging)

Router mounting:
• app.use('/products', productRouter) — public, no auth
• app.use('/cart', authenticateJWT, cartRouter) — auth required
• app.use('/checkout', authenticateJWT, rateLimiter, checkoutRouter) — auth + rate limit
• app.use('/admin', authenticateJWT, requireRole('admin'), adminRouter)

Service layer for checkout: CheckoutService handles cart validation, inventory check, pricing calculation, payment gateway call, order creation, email trigger — all independently testable.

Error handling: All async handlers wrapped in try/catch → next(err) → centralized error middleware last.

Config: All env vars validated at startup; secrets fetched from AWS Secrets Manager if production.
Expected mistake: Putting business logic in route handlers, no service layer, global auth applied to public routes. The key differentiator is knowing when to add middleware per-router vs app-level, and knowing when a service layer is necessary.
