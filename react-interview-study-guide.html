Core JavaScript Fundamentals

Beginner → Intermediate
1
Library vs Framework — what is the core difference?
Beginner
▼
Definition
A library is code you call; a framework calls your code. The key concept is Inversion of Control.
Deep Explanation
With a library (like lodash, axios), you control the program flow. You import it and call its functions whenever you want.

With a framework (like Angular, Next.js), the framework owns the control flow. It calls your code at prescribed points (lifecycle hooks, route handlers). This is the Hollywood Principle: "Don't call us, we'll call you."

Framework = opinionated scaffolding + enforced conventions. Library = focused utility, you integrate it however you want.
Simple Example
// Library: YOU call the function import axios from 'axios'; axios.get('/api/data'); // You decide when + how // Framework: IT calls YOUR function // Angular's lifecycle hook — Angular calls this, not you ngOnInit() { this.loadData(); }
Common Pitfalls
Saying "React is a framework" — React is a UI library. It only handles the View layer.
Confusing "opinionated" with "framework" — a library can be opinionated too.
Interview One-Liner
"In a library, you call the code. In a framework, the framework calls your code — this is Inversion of Control."
Interviewer Follow-Up

Q: "So if React is just a library, what makes a 'React framework'?"

Expected answer: Next.js, Remix, Gatsby — they add routing, SSR, file-based conventions ON TOP of React, making React + framework = full-stack framework.

Is React really a library or a framework? ⭐
⭐ ImportantBeginner
▼
Definition
React is officially a JavaScript library for building user interfaces. It handles only the View layer (V in MVC).
Deep Explanation
React by itself is a library — it only renders UI. It has no built-in routing, no global state management, no HTTP client, no form validation.

However, the React ecosystem (React + Router + Redux + etc.) behaves like a framework. That's why people colloquially call it a "framework."

Interview trap: The question is testing whether you know the precise definition and whether you can back it up. Don't just say "it's a framework because everyone uses it that way."

What is an SPA? Advantages and disadvantages? ⭐
⭐Beginner
▼
Definition
A Single Page Application loads one HTML page and dynamically updates content via JavaScript without full page reloads. Navigation is handled client-side.
Deep Explanation
In a traditional MPA (Multi-Page App), every route change triggers a new HTML file from the server. In an SPA, the server sends one HTML file once; React's router intercepts navigation and swaps components in the DOM.

This gives a native app feel — no white flash between pages, shared state persists across routes, and assets (CSS, JS) only load once.
Pros vs Cons
AdvantagesDisadvantages
Fast, seamless navigation after initial loadSlow initial load (large JS bundle)
Rich, app-like UXPoor SEO by default (empty HTML)
Reduced server loadRequires JS enabled
Shared state across viewsComplex state management
Works offline with PWABrowser history management needed
Interview One-Liner
"An SPA loads once, then React handles routing client-side — great UX but bad SEO unless you use SSR (like Next.js)."
Interviewer Follow-Up

Q: "How would you solve the SEO problem of SPAs?"

Answer: Server-Side Rendering (Next.js), Static Site Generation, or pre-rendering. The server sends pre-rendered HTML for crawlers to read.

What is Vite? Why is it faster than CRA?
Beginner
▼
Definition
Vite is a next-generation build tool that leverages native ES modules in the browser for dev, and Rollup for production builds. CRA uses Webpack for everything.
Deep Explanation
Why CRA is slow: CRA bundles your entire app with Webpack before the dev server can respond. For 1000 files, Webpack parses, transforms, and bundles ALL of them on every startup.

Why Vite is fast:
1. No bundling in dev — Vite serves files via native ESM. The browser imports files on-demand. Startup time is near-instant regardless of project size.
2. esbuild for transforms — esbuild (Go-based) transpiles TypeScript/JSX 10-100x faster than Babel.
3. Hot Module Replacement (HMR) — Only the changed module is updated, not the whole bundle.

What is the Virtual DOM? How does React use it? ⭐
⭐Intermediate
▼
Definition
The Virtual DOM (VDOM) is a lightweight in-memory JavaScript representation of the real DOM. React uses it to compute the minimal set of changes needed before touching the actual DOM.
Deep Explanation
Real DOM manipulation is expensive — browsers must recalculate styles, layout, and repaint. React avoids unnecessary DOM operations with a 3-step process:

1. Render phase: When state changes, React creates a new VDOM tree.
2. Diffing (Reconciliation): React compares the new VDOM with the previous one (the "diff"). It identifies only what changed.
3. Commit phase: React applies only the minimal DOM mutations needed.

Key: The VDOM itself isn't magic — calling DOM APIs directly can be faster. The VDOM's value is in batching updates and giving React a declarative API where you describe UI by state, not by manual DOM imperatives.
Simple Example
// Without VDOM: manual DOM, expensive document.getElementById('count').textContent = count; // With React VDOM: declarative, React handles the diff const App = () => <p>{count}</p>; // React figures out only the text node changed
Common Pitfalls
Saying "VDOM is always faster than real DOM" — it's not. Direct DOM can be faster; VDOM's benefit is developer ergonomics + smart batching.
Confusing VDOM with Shadow DOM (browser native, for web components).
Interview One-Liner
"The VDOM is React's in-memory DOM copy. On state change, React diffs old vs new VDOM and only updates what changed in the real DOM — minimizing expensive repaints."
Interviewer Trap
Trap: "Is the Virtual DOM always faster?"
→ No! Svelte (no VDOM) and SolidJS are often faster because they compile to precise DOM updates. The VDOM is a useful abstraction, not a performance guarantee.

Explain React's Reconciliation (diffing) algorithm. How do keys affect it? ⭐
⭐Intermediate
▼
Definition
Reconciliation is React's algorithm for computing the diff between the current and new VDOM trees, then updating the real DOM minimally. It uses two heuristics to achieve O(n) time instead of the theoretical O(n³).
Deep Explanation
Heuristic 1 — Type check: If the root element type changes (e.g. <div> → <span>), React tears down the entire subtree and builds fresh. If the type is the same, React updates attributes in-place.

Heuristic 2 — Keys: When rendering lists, React uses keys to match old vs new elements. Without keys, React assumes position-based matching — adding an element at the top causes every other element to be re-rendered. With stable keys, React knows exactly which element moved, was added, or removed.

Fiber: Modern React uses Fiber architecture — reconciliation is async and interruptible, allowing React to pause work and prioritize urgent updates (user input > background data load).
Keys Example
// BAD: index as key — list reorder causes full re-render {items.map((item, i) => <Item key={i} {...item} />)} // GOOD: stable unique key {items.map(item => <Item key={item.id} {...item} />)}
Common Pitfalls
Using Math.random() as key — generates new key every render, forcing complete re-mount.
Using array index for lists that can be reordered, filtered, or items added/removed at front.
Interview One-Liner
"React uses two heuristics — type-check and keys — to reduce diffing from O(n³) to O(n). Keys tell React which element is which across renders."
5
What does JSX compile to?
Beginner
▼
Definition
JSX is syntactic sugar that Babel compiles to React.createElement() calls (or in React 17+, the new JSX transform using _jsx() from react/jsx-runtime).
Example
// What you write (JSX): const el = <h1 className="title">Hello</h1>; // What Babel compiles to (old transform): const el = React.createElement( 'h1', { className: 'title' }, 'Hello' ); // React 17+ new JSX transform (no React import needed): import { jsx as _jsx } from 'react/jsx-runtime'; const el = _jsx('h1', { className: 'title', children: 'Hello' });
Interview One-Liner
"JSX compiles to React.createElement() calls. It's not magic — it's just a more readable way to write JavaScript object trees."
Interviewer Trap
Trap: "Why did older React need import React from 'react' in every file?"
→ Because JSX compiled to React.createElement(), React had to be in scope. The new JSX transform (React 17+) auto-imports the jsx runtime, removing this requirement.

React State & Props

Beginner → Advanced
1
What is useState? What triggers a re-render? ⭐
⭐Beginner
▼
Definition
useState is a hook that adds local state to a functional component. Calling the setter function schedules a re-render with the new state value.
Deep Explanation
React triggers a re-render when:
1. setState (useState setter) is called with a new value
2. Parent re-renders and passes new props
3. Context value changes
4. useReducer dispatch is called

Important: React uses Object.is() to compare old vs new state. If you set state to the same value (or same object reference), React skips the re-render (bail-out optimization).
Example
const [count, setCount] = useState(0); // Triggers re-render (new value) setCount(count + 1); // Does NOT trigger re-render (same value) setCount(0); // if count is already 0 // Object trap — same reference, no re-render! const [obj, setObj] = useState({ a: 1 }); obj.a = 2; setObj(obj); // SAME reference → no re-render! setObj({ ...obj }); // NEW reference → re-render ✓
Common Pitfalls
Mutating state directly and re-setting same reference — React won't re-render.
Reading state immediately after setState — state updates are async/batched.
Interview One-Liner
"useState returns a state value and a setter. Calling the setter with a new value (by Object.is) schedules a re-render. Mutating objects without new references silently breaks reactivity."
2
What is a stale closure in useEffect? How to fix it? ⭐
⭐Intermediate
▼
Definition
A stale closure occurs when a useEffect callback (or event listener inside it) captures a variable from its initial render and never sees updated values — because the effect didn't re-run.
Deep Explanation
JavaScript closures capture variables by reference at the time the function is created. When useEffect runs, it creates a closure over the current values. If dependencies aren't listed, the effect's closure holds the old values forever — even if state updates.

Fixes:
1. Add the variable to dependency array — effect re-runs when it changes
2. Use functional setState: setCount(prev => prev + 1) — doesn't need current count in closure
3. useRef — ref.current always gives the latest value without causing re-renders
Example
// STALE CLOSURE BUG const [count, setCount] = useState(0); useEffect(() => { const id = setInterval(() => { console.log(count); // Always logs 0! }, 1000); return () => clearInterval(id); }, []); // [] means count is captured at mount (stale) // FIX 1: Add to deps }, [count]); // Re-runs when count changes // FIX 2: Functional update (no closure needed) setCount(prev => prev + 1);
Interview One-Liner
"A stale closure is when useEffect captures old state because it ran with an empty dep array. Fix: add the variable to deps, or use functional setState / useRef for latest value."
3
What is the race condition in React data fetching? How to prevent it? ⭐
⭐Advanced
▼
Definition
A race condition occurs when multiple async fetches are triggered in sequence (e.g. user types fast), and a slower earlier response overwrites the result of a faster later response.
Deep Explanation
User types "a" → fetch for "a" starts (slow, 800ms)
User types "ab" → fetch for "ab" starts (fast, 200ms)
Fetch "ab" resolves first (200ms), state = results for "ab"
Fetch "a" resolves later (800ms), state = results for "a" ← WRONG!

The solution is to use a cleanup flag or AbortController to ignore stale responses.
Fix with ignore flag
useEffect(() => { let ignore = false; async function fetchData() { const data = await fetch(`/api/search?q=${query}`); if (!ignore) setResults(data); } fetchData(); return () => { ignore = true; }; // cleanup }, [query]);
With AbortController (modern)
useEffect(() => { const controller = new AbortController(); fetch(`/api?q=${query}`, { signal: controller.signal }) .then(r => r.json()).then(setResults) .catch(e => { if (e.name !== 'AbortError') throw e; }); return () => controller.abort(); }, [query]);
Interview One-Liner
"Use an ignore flag or AbortController in useEffect cleanup — if a new render triggered a new fetch, mark previous responses as stale and ignore them."
4
What is prop drilling? How does Context solve it? ⭐
⭐Intermediate
▼
Definition
Prop drilling is passing props through multiple intermediate components that don't need them, just to get data to a deeply nested child. Context API provides a way to share values globally without drilling.
Example
// Problem: theme drills through A → B → C App (theme) → ComponentA (theme) → ComponentB (theme) → ComponentC uses theme // Solution: Context const ThemeCtx = React.createContext('light'); // Provider at top <ThemeCtx.Provider value={theme}><App /></ThemeCtx.Provider> // Consumer anywhere deep const theme = useContext(ThemeCtx); // No drilling!
Performance Implication
Context has a key performance issue: every consumer re-renders when the context value changes, even if the specific part they use didn't change.

Fix strategies: Split context into smaller contexts, use useMemo on the context value, or use a dedicated state library like Zustand/Redux for high-frequency updates.
Interview One-Liner
"Prop drilling passes data through components that don't use it. Context solves this but causes all consumers to re-render on any change — use carefully for high-frequency state."
5
Why use functional updates: setCount(prev => prev + 1)?
Intermediate
▼
Definition
Functional updates pass a function to the setter that receives the latest state as its argument, ensuring you never operate on stale state — regardless of batching or closures.
Why it matters
// BUG: multiple updates in same event — both read count=0 setCount(count + 1); // 0 + 1 = 1 setCount(count + 1); // 0 + 1 = 1 (still reads 0!) // Final count: 1 (expected: 2) // FIX: functional update — each receives latest setCount(prev => prev + 1); // 0 → 1 setCount(prev => prev + 1); // 1 → 2 ✓
Interview One-Liner
"Use functional updates when new state depends on old state — avoids stale closure bugs and is safe with React's automatic batching."

What is useRef? How is it different from useState? ⭐
⭐Beginner
▼
Definition
useRef returns a mutable object { current: value } that persists across renders but does NOT trigger a re-render when changed. It's used for DOM references and mutable instance variables.
useRef vs useState
useRefuseState
Triggers re-render?NOYES
Persists across renders?YESYES
Mutable?YES (ref.current)Only via setter
Use caseDOM refs, timers, prev valuesUI state
Common Uses
// 1. DOM reference const inputRef = useRef(null); <input ref={inputRef} /> inputRef.current.focus(); // programmatic focus // 2. Store interval ID (no re-render needed) const timerRef = useRef(null); timerRef.current = setInterval(cb, 1000); clearInterval(timerRef.current); // 3. Interview One-Liner
"useRef is a mutable box — changing ref.current doesn't trigger re-renders. Use it for DOM access, storing timers, and solving stale closure issues."
2
When to use useReducer instead of useState? ⭐
⭐Intermediate
▼
Definition
useReducer manages state via a reducer function — a pure function that takes (state, action) and returns new state. Best for complex state logic, multiple sub-values, or when next state depends on previous.
When to Choose
Use useReducer when:
- State has multiple sub-fields that update together
- State transitions have names/types (ADD_TODO, REMOVE_TODO)
- Complex conditionals in your setState logic
- You want testable, predictable state transitions
- Next state depends on previous in complex ways

Use useState when: state is a simple value, or 2-3 independent values.
Counter with useReducer
const reducer = (state, action) => { switch (action.type) { case 'increment': return { count: state.count + 1 }; case 'decrement': return { count: state.count - 1 }; case 'reset': return { count: 0 }; default: return state; } }; const [state, dispatch] = useReducer(reducer, { count: 0 }); dispatch({ type: 'increment' });
Interview One-Liner
"Use useReducer when state logic is complex, has named transitions, or multiple sub-values — it's like a local Redux, making state changes predictable and testable."
Interviewer Follow-Up

Q: "How does useReducer relate to Redux?"

Same mental model: reducer + dispatch + actions. Redux adds a global store, middleware, DevTools, and cross-component sharing. useReducer is component-local.
3
Difference between useMemo and useCallback? ⭐
⭐Intermediate
▼
Definition
useMemo memoizes a computed value. useCallback memoizes a function reference. Both recompute only when dependencies change.
Comparison
useMemouseCallback
ReturnsMemoized valueMemoized function
Use caseExpensive calculationsStable function refs for child props
EquivalentuseMemo(() => fn(), deps)useMemo(() => fn, deps)
Example
// useMemo: heavy computation cached const sortedList = useMemo( () => items.sort((a, b) => a.price - b.price), [items] ); // useCallback: stable reference for child component const handleClick = useCallback(() => { dispatch({ type: 'ADD', payload: item }); }, [item, dispatch]); // Without useCallback, new fn reference each render // → React.memo child re-renders unnecessarily
Common Pitfalls
Using useMemo/useCallback everywhere "for performance" — they have overhead too. Only use when you measure a problem.
useMemo without React.memo on child — memoized function is useless if child always re-renders anyway.
Interview One-Liner
"useMemo caches a computed value; useCallback caches a function reference. Both prevent unnecessary recalculation/recreation — but only pay off when combined with React.memo or when computation is truly expensive."
4
What is referential equality and why does it matter? ⭐⭐
Intermediate
▼
Definition
Referential equality means two variables point to the exact same object in memory (===). In React, new object/function literals created on every render always fail equality checks, causing unnecessary re-renders.
Why it breaks React.memo
const Parent = () => { // New function object EVERY render const onClick = () => doSomething(); return <Child onClick={onClick} />; }; // React.memo on Child compares: onClick_old === onClick_new // → false every time → memo is useless! // FIX: useCallback gives stable reference const onClick = useCallback(() => doSomething(), []);
Interview One-Liner
"Every render creates new objects/functions — they're always 'not equal' by reference. useCallback and useMemo preserve stable references, making React.memo and useEffect deps work correctly."

How to implement a protected route in React Router v6? ⭐
⭐Intermediate
▼
Definition
A protected route redirects unauthenticated users to a login page. In v6, this is done via a wrapper component that checks auth status and renders either <Outlet /> or <Navigate />.
Implementation
// ProtectedRoute.jsx const ProtectedRoute = () => { const { isAuthenticated } = useAuth(); return isAuthenticated ? <Outlet /> : <Navigate to="/login" replace />; }; // App.jsx routing setup <Routes> <Route element={<ProtectedRoute />}> <Route path="/dashboard" element={<Dashboard />} /> <Route path="/profile" element={<Profile />} /> </Route> <Route path="/login" element={<Login />} /> </Routes>
Real-World: Redirect back after login
// Save intended destination before redirect <Navigate to="/login" state={{ from: location }} replace /> // In Login component, redirect after success const { state } = useLocation(); navigate(state?.from?.pathname || '/');
Interview One-Liner
"In v6, wrap protected routes in a layout route component that renders Outlet if authenticated, or Navigate to /login if not."

What is Outlet? How is it used for nested routes?
Intermediate
▼

<Outlet /> is a placeholder in a parent route component that renders the matched child route's element. It's how v6 handles nested routing and shared layouts.

"Outlet renders the active child route inside a layout component — it's how v6 composes shared headers/sidebars with nested pages."

Redux / Redux Toolkit

Intermediate → Advanced
1
What is Redux? What problem does it solve? ⭐
⭐Intermediate
▼
Definition
Redux is a predictable state container for JavaScript apps. It solves the problem of sharing state across many components with a single global store and strict unidirectional data flow.
Core Concepts
Three principles:
1. Single source of truth — one global store, predictable state
2. State is read-only — only dispatch actions to change state
3. Changes via pure reducers — no side effects, testable

When to use Redux vs Context:
Context: simple, low-frequency shared state (theme, user)
Redux: complex, high-frequency, cross-feature state; dev tools needed; large team
Data Flow
Component → dispatch(action) → reducer(state, action) → new state → store → component re-renders
Interview One-Liner
"Redux provides a single predictable global store where state changes happen only through dispatched actions processed by pure reducers — making state changes traceable and testable."
2
What is a slice in Redux Toolkit? How does it simplify Redux? ⭐
⭐Intermediate
▼
Definition
A slice is a collection of Redux reducer logic and actions for a single feature. createSlice auto-generates action creators and action types from reducer functions — eliminating 90% of Redux boilerplate.
Before vs After RTK
// BEFORE (classic Redux — 40+ lines) const INCREMENT = 'counter/increment'; const increment = () => ({ type: INCREMENT }); const counterReducer = (state = 0, action) => { ... }; // AFTER RTK createSlice (10 lines!) const counterSlice = createSlice({ name: 'counter', initialState: { value: 0 }, reducers: { increment: state => { state.value += 1; }, // Immer! decrement: state => { state.value -= 1; }, setBy: (state, action) => { state.value = action.payload; } } }); export const { increment, decrement, setBy } = counterSlice.actions;
Immer Under the Hood
RTK uses Immer inside reducers — you can write "mutating" code (state.value += 1) and Immer produces a new immutable state behind the scenes. This removes the need for ...spread gymnastics in every reducer.
Interview One-Liner
"createSlice auto-generates action creators + action types from reducer functions, and Immer lets you write mutating-looking code that's actually immutable."
3
What is createAsyncThunk? How does it handle loading/error states? ⭐
⭐Advanced
▼
Definition
createAsyncThunk generates a thunk that dispatches lifecycle actions: pending, fulfilled, and rejected — automatically, based on your async function's promise.
Full Example
// 1. Create the thunk export const fetchUser = createAsyncThunk( 'user/fetchById', async (userId, { rejectWithValue }) => { try { const res = await axios.get(`/api/users/${userId}`); return res.data; } catch (err) { return rejectWithValue(err.response.data); } } ); // 2. Handle in slice extraReducers: (builder) => { builder .addCase(fetchUser.pending, state => { state.status = 'loading'; }) .addCase(fetchUser.fulfilled, (state, action) => { state.status = 'succeeded'; state.user = action.payload; }) .addCase(fetchUser.rejected, (state, action) => { state.status = 'failed'; state.error = action.payload; }); }
Interview One-Liner
"createAsyncThunk wraps async logic and auto-dispatches pending/fulfilled/rejected actions — use extraReducers in your slice to handle each lifecycle state."
4
Explain Flux architecture. How does Redux implement it? What role does Immer play? ⭐
⭐Advanced
▼
Definition
Flux is an application architecture pattern for unidirectional data flow: Action → Dispatcher → Store → View. Redux is an implementation of Flux (simplified — no dispatcher, reducer is pure function).
Flux vs Redux
ConceptFluxRedux
StoresMultipleSingle store
DispatcherExplicit dispatcherBuilt into store.dispatch
State mutationStores handle mutationPure reducers (immutable)
Data flowUnidirectionalUnidirectional
Immer's Role
Immutability is core to Redux — reducers must return new state objects, not mutate the old one. Writing { ...state, user: { ...state.user, name: action.payload } } gets verbose fast.

Immer creates a "draft" proxy of state. You mutate the draft, Immer produces a structurally-shared immutable copy. RTK uses Immer automatically in all slice reducers.
Interview One-Liner
"Redux implements Flux's unidirectional data flow — Action → Reducer → Store → View. Immer makes immutable updates ergonomic by letting you write mutating code that produces immutable state."

⭐Beginner

What are Axios interceptors? Give a real use case. ⭐
⭐Intermediate
▼
Definition
Interceptors are middleware functions that run before every request is sent or after every response is received. They can modify, log, or handle errors globally.
Real-World: Auth Token Injection
// Request interceptor: add token to every request axios.interceptors.request.use(config => { const token = localStorage.getItem('accessToken'); if (token) config.headers.Authorization = `Bearer ${token}`; return config; }); // Response interceptor: handle 401 globally axios.interceptors.response.use( response => response, async error => { if (error.response?.status === 401) { await refreshToken(); return axios.request(error.config); // retry } return Promise.reject(error); } );
Interview One-Liner
"Interceptors are request/response middleware. Common use: attach auth tokens on request, handle 401/token refresh on response — once, globally, without repeating in every API call."
3
How would you implement automatic token refresh with interceptors? ⭐⭐
Advanced
▼
The Challenge
When an access token expires, you need to:
1. Detect the 401 error
2. Call the refresh endpoint once (not multiple times if concurrent requests fail)
3. Retry all failed requests with the new token

The tricky part: if 5 requests fail simultaneously, you only want ONE refresh call. Use a promise queue to hold subsequent requests while the first refresh happens.
Production Implementation
let isRefreshing = false; let failedQueue = []; const processQueue = (error, token = null) => { failedQueue.forEach(({ resolve, reject }) => error ? reject(error) : resolve(token) ); failedQueue = []; }; api.interceptors.response.use(null, async error => { if (error.response?.status !== 401) throw error; if (isRefreshing) { // Queue this request until token refreshed return new Promise((resolve, reject) => { failedQueue.push({ resolve, reject }); }).then(token => { error.config.headers.Authorization = `Bearer ${token}`; return api(error.config); }); } isRefreshing = true; try { const { data } = await api.post('/auth/refresh'); processQueue(null, data.token); error.config.headers.Authorization = `Bearer ${data.token}`; return api(error.config); } catch (e) { processQueue(e, null); throw e; } finally { isRefreshing = false; } });
Interview One-Liner
"Use a lock flag and a queue — first 401 triggers refresh, subsequent 401s wait in queue. When refresh resolves, drain the queue and retry all failed requests."

Mock Interview — 5 Tricky Questions

Simulate real interview pressure
Answer each question mentally first, then click "Reveal Expert Answer" to compare. These questions are designed to trip up even experienced developers.
Interviewer Question 1 of 5 — Intermediate
"You call setCount(count + 1) three times in a row inside a click handler. What will count be after the click? Why?"
Answer: Count increases by only 1, not 3. In React 18+ (auto-batching), all three setCount calls in the same event handler are batched into one re-render. Since all three read the same stale value of count (e.g. 0), they all compute 0+1=1. Only the last update wins.

Fix: Use functional updates: setCount(prev => prev + 1) three times — each receives the latest queued state, so count correctly becomes 3.
Trap: Many candidates say "it will be count + 3" — they forget about batching and stale reads.
Interviewer Question 2 of 5 — Advanced
"I have a context providing an object: {'{'} user, theme {'}'}. A deeply nested component only uses theme. If only user changes, does the theme component re-render? How would you prevent it?"
Yes, it re-renders. Every context consumer re-renders when the context value changes — React doesn't know which part of the value changed. Since {'{'} user, theme {'}'} is a new object reference every time, even if theme is unchanged, all consumers re-render.

Fixes:
1. Split contexts: separate UserContext and ThemeContext — theme component only subscribes to ThemeContext.
2. Memoize value: useMemo(() => ({'{'} user, theme {'}'}), [user, theme]) — prevents new object reference if neither changed.
3. Use Zustand/Redux for user; keep ThemeContext simple.
Trap: Candidates often say "React will bail out if theme didn't change" — wrong. Context doesn't do granular diffing.
Interviewer Question 3 of 5 — Advanced
"When can useMemo actually hurt performance? Give a concrete example."
useMemo has overhead: React must store the memoized value, compare dependencies on every render, and run the comparison logic. If the computation being memoized is trivial (e.g. items.length), the memo overhead exceeds the computation cost.

Example: const count = useMemo(() => items.length, [items]) is slower than const count = items.length. Array length access is O(1); useMemo comparison is also O(1) but adds indirection.

Rule: Only memoize when: (1) computation is demonstrably slow (measure with profiler), or (2) result is passed as prop to a React.memo child. Otherwise, the cost of memoization outweighs the benefit.
Trap: "More memoization = better performance" is a common misconception. Premature optimization is a real anti-pattern.
Interviewer Question 4 of 5 — Advanced Redux
"In RTK slice reducers, you write state.user.name = action.payload. Isn't that mutating state? Won't Redux break?"
Great question — this is Immer magic. RTK wraps all slice reducers with Immer's produce() function. When your reducer runs, it doesn't receive the real state object — it receives an Immer "draft proxy." Any mutations on the draft are recorded, and Immer produces a brand-new immutable state object at the end.

So you appear to mutate, but Immer ensures the output is a completely new object with structural sharing (unchanged parts are the same reference). Redux receives a new reference → triggers re-renders correctly.

Caveat: Outside of createSlice/createReducer (e.g. in plain reducer), you must NOT mutate — Immer isn't wrapping it.
Trap: This trips up candidates who know "Redux requires immutability" but don't know how RTK/Immer works under the hood.
Interviewer Question 5 of 5 — System Design
"You're building a large e-commerce app. Product listings, cart, user auth, search filters. Would you use Context API or Redux? Walk me through your architecture decision."
Hybrid approach — this is the senior answer:

Redux (RTK) for: Cart (complex, cross-feature, frequently updated), User auth (token, permissions — needed everywhere), Product listings (server state, pagination, caching)

Context for: Theme/locale (low-frequency, rarely changes), Modal/drawer state (UI-only, not business logic)

Consider React Query/TanStack Query for: Product listings, search results — it handles caching, loading states, background refetch, and stale-while-revalidate out of the box. This removes the need for async thunks for server state entirely.

Rule of thumb: Redux for client state (cart, UI state shared across features). React Query for server/remote state. Context for rarely-changing global config.
Expected mistake: Saying "just use Redux for everything" or "just use Context for everything" without justification. Interviewers want to see trade-off thinking.
